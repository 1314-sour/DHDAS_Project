# 任务二分支合并说明

本分支完成任务二：工程管理应用，并且已经与任务一的通道管理能力、任务二要求的存储切换链路完成联动。  
需要特别说明的是：本分支不是“只新增一个工程管理插件”，而是同时包含 `Contracts`、`Service`、`Shell DI`、`StorageService` 和插件层的必要改动，因此合并时应按模块整体保留。

这些契约统一放在 `DHDAS.Contracts`，供工程管理插件、后端服务、存储节点共享，不把业务状态写死在 UI 内部。

这意味着工程管理前端本身就是可插拔 DLL，基础版软件后续可以通过投放插件 DLL 的方式扩展功能。

## 当前实际实现的目录与数据结构

当前分支采用的工程结构如下：

```text
{工程根目录}/
  project.json
  Config/
    channels.json
  Experiments/
    {ExperimentId}/
      experiment.json
      Config/
        channels.json
      RawData/
        recorded_data.dat
      Results/
```

说明：

- 工程级只保留 `Config` 和 `Experiments`
- `RawData`、`Results` 已调整为“实验级目录”，不再挂在工程根目录下
- 存储文件的实际写入目标为 `{工程}/Experiments/{实验ID}/RawData/recorded_data.dat`

## 与通道管理、存储链路的联动方式

### 1. 与通道管理的联动

工程管理不是直接依赖 `ChannelManager` 插件 UI，而是依赖任务一提供的契约和服务：

- `ChannelInfo`
- `ChannelSettingsChangedMessage`
- `IInstrumentService`
- `ChannelHardwareProfile`

当前联动方式是：

```text
通道管理/仪器服务
  -> IMessenger.Send(ChannelSettingsChangedMessage)
工程管理服务
  -> IMessenger.Listen<ChannelSettingsChangedMessage>()
  -> 保存当前工程/实验的 channels.json
```

当前分支已经做了两点关键适配：

- 创建工程时，会把“当前有效通道配置”写入 `Config/channels.json`
- 持久化时只写入启用通道，不再把已关闭通道全部落盘

打开工程时会：

- 读取工程 `Config/channels.json`
- 基于 `ChannelHardwareProfile` 还原 2048 通道运行时快照
- 未出现在配置文件中的通道按“默认关闭”处理
- 再通过 `IInstrumentService.ApplySettings(...)` 恢复到运行时

因此，任务二当前实现实际上依赖任务一已经存在；如果单独把任务二合到一个不含任务一能力的 `main`，会因为缺少通道契约和仪器服务而无法完整工作。

### 2. 与存储服务的联动

`StorageService` 当前通过订阅 `ProjectChangedMessage` 感知工程/实验切换：

```text
ProjectService
  -> IMessenger.Send(ProjectChangedMessage)
StorageService
  -> 切换/关闭 recorded_data.dat 文件句柄
```

当前行为如下：

- 打开工程并激活实验后，存储目标自动切到当前实验目录
- 创建实验并切换后，存储文件句柄自动切换
- 关闭工程、删除工程、删除当前实验时，存储句柄会先关闭，避免文件锁

当前 `Program.cs` 中的本地集成流水线为：

```text
AcquisitionService -> DataPushService -> StorageService
```

这条链路已经满足任务二“切换工程时切换落盘目标”的验证需求。

## 合并注意事项

- `Program.cs` 中要同时保留 `AddChannelManagerModule()` 和 `AddProjectManagerModule()`
- `AddProjectManagerModule()` 应继续作为工程管理统一 DI 入口，不建议在 Shell 中零散 `AddSingleton`
- `StorageService` 对 `ProjectChangedMessage` 的订阅必须保留，否则工程/实验切换后不会切换落盘目标
- 若任务三/任务四后续也要感知实验切换，应继续复用 `ExperimentSwitchedMessage` 与 `ISignalProcessingContext`
- 若其他分支改过目录结构，最终应以“`RawData` / `Results` 位于实验级目录”这一版为准
- 若其他分支也改了通道配置落盘逻辑，最终应保留“只保存启用通道、运行时补全关闭通道”的实现
- 如果任务四要重组全量流水线，可以调整 `pipelineScheme`，但不要去掉 `StorageService` 对工程切换消息的响应逻辑
