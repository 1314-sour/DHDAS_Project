# 分布式管理应用需求对照与整改方案

## 1. 结论摘要

当前实现已经具备“单机双开模拟发送端/接收端、手动生成正弦波、TCP 发送、ACK 反馈、接收端显示波形”的演示闭环，但它更接近“基础通信链路 Demo”，尚未完全达到任务四“分布式管理应用（Distribution Manager）”的规约要求。

总体判断：

| 模块 | 当前状态 | 说明 |
| --- | --- | --- |
| UI 界面 | 部分满足 | 已有发送端/接收端角色区分和基础路由按钮，但未实现“计算节点映射表”。 |
| ViewModel 逻辑 | 部分满足 | 已能触发路由更新、生成波形、发送当前波形，但没有连接/断开指令、实时流量、丢包统计。 |
| 契约定义 | 部分满足 | `NetworkDataPacket` 已存在且基于 MessagePack；缺少 `INetworkService`，`NetworkRoute` 字段不足。 |
| 后端服务 | 部分满足 | 已有 `NetworkSenderNode`/`NetworkReceiverNode` 和 TCP 传输；发送逻辑不是高性能异步 Buffer 队列模式。 |
| 联动需求 | 未满足 | 网络异常当前通过 `IDistributedFeedbackService` 弹窗，不是通过 `IMessenger` 通知工程管理应用红灯告警。 |
| 工业级约束 | 部分满足 | 有基础异常处理；DI 注册、日志接口、模块化注册未满足。 |

## 2. 当前已实现的能力

当前代码中已经实现或部分实现了以下能力：

1. 单机双角色运行：
   - `--role sender`
   - `--role receiver`

2. 发送端能力：
   - 手动生成 1000 点、5 个完整周期的正弦波。
   - 生成后通过 `IWaveformSnapshotService` 在发送端“实时波形显示”中展示。
   - 手动点击“发送当前波形”后再发送，不自动轮询。

3. 接收端能力：
   - `NetworkReceiverNode` 监听 TCP `5000` 端口。
   - 收到数据后还原为 `RawDataPacket`。
   - 通过 `IWaveformSnapshotService` 在接收端“实时波形显示”中展示完整波形。
   - 返回 `NetworkAckPacket` 给发送端确认接收结果。

4. 网络协议：
   - `NetworkDataPacket` 使用 MessagePack 序列化。
   - 当前传输增加了 4 字节长度头，解决 TCP 流式读取边界问题。

5. 反馈机制：
   - 使用 `IDistributedFeedbackService` 在前端弹窗显示发送成功、连接失败、接收成功等反馈。

## 3. 任务四逐条对照

### 3.1 UI 界面：计算节点映射表

原需求：

> 实现“计算节点映射表”，包含：目标节点名称、IP地址、端口号、负责转发的通道范围（如：1-10号通道发给 A 电脑）。

当前状态：**部分满足**

当前已有：

- `DHDAS.Plugin.NetworkConfig` 插件。
- 发送端/接收端模式区分。
- 发送端可以添加本地回环路由。

当前不足：

- 没有表格形式的“计算节点映射表”。
- 没有目标节点名称字段。
- 端口号当前固定为 `5000`。
- 通道只支持单个 `ChannelId`，不支持通道范围，如 `1-10`。
- 没有新增、删除、编辑多条路由的完整 UI。

建议修改：

1. 将当前按钮式 UI 改为表格式 UI，例如 `DataGrid`。
2. 新增路由模型字段：

```csharp
public record NetworkRoute(
    string NodeName,
    string TargetIp,
    int Port,
    int StartChannelId,
    int EndChannelId);
```

3. UI 表格列建议：

| 列名 | 类型 | 示例 |
| --- | --- | --- |
| 节点名称 | TextBox | A 电脑 |
| IP 地址 | TextBox | 192.168.1.20 |
| 端口号 | Numeric/TextBox | 5000 |
| 起始通道 | Numeric/TextBox | 1 |
| 结束通道 | Numeric/TextBox | 10 |
| 启用 | CheckBox | true |
| 操作 | Button | 连接/断开/删除 |

4. 保留当前“生成测试正弦波”和“发送当前波形”作为调试区，但不要替代正式映射表。

## 4. ViewModel 逻辑对照

原需求：

> 处理连接/断开指令，展示各网络链路的实时流量（MBps）和丢包统计。

当前状态：**部分满足**

当前已有：

- `NetworkConfigViewModel.AddRoute()` 可以更新本地路由。
- `GenerateWaveform()` 和 `SendOnce()` 可以完成测试数据生成与发送。

当前不足：

- 没有连接/断开命令。
- 没有链路状态模型。
- 没有实时流量 MBps 统计。
- 没有丢包统计。
- 没有每个远端节点的在线/离线状态。

建议修改：

1. 新增链路状态模型：

```csharp
public class NetworkLinkStatus
{
    public string NodeName { get; set; } = string.Empty;
    public string Endpoint { get; set; } = string.Empty;
    public bool IsConnected { get; set; }
    public double Mbps { get; set; }
    public long SentPackets { get; set; }
    public long AckPackets { get; set; }
    public long FailedPackets { get; set; }
    public double PacketLossRate { get; set; }
    public string LastError { get; set; } = string.Empty;
}
```

2. ViewModel 增加：

```csharp
ObservableCollection<NetworkRoute> Routes
ObservableCollection<NetworkLinkStatus> LinkStatuses
ICommand ConnectCommand
ICommand DisconnectCommand
ICommand ApplyRoutingCommand
```

3. 后端定期发布状态快照，ViewModel 订阅后刷新 UI。

## 5. 契约定义对照

### 5.1 INetworkService

原需求：

> 定义 `INetworkService`：提供 `UpdateRoutingTable(List<NetworkRoute>)` 接口。

当前状态：**未满足**

当前已有：

- `NetworkSenderNode.UpdateConfig(IEnumerable<NetworkRoute>)`

当前不足：

- 没有 `INetworkService` 契约接口。
- UI 插件直接依赖 `NetworkSenderNode`，导致应用层和服务节点耦合过重。

建议修改：

新增契约：

```csharp
public interface INetworkService
{
    void UpdateRoutingTable(List<NetworkRoute> routes);
    Task ConnectAsync(NetworkRoute route, CancellationToken ct = default);
    Task DisconnectAsync(NetworkRoute route, CancellationToken ct = default);
    IObservable<NetworkLinkStatus> LinkStatusChanged { get; }
}
```

然后：

- `NetworkSenderNode` 实现 `INetworkService`，或新增 `NetworkService` 聚合发送节点。
- `NetworkConfigViewModel` 依赖 `INetworkService`，不直接依赖 `NetworkSenderNode`。

### 5.2 NetworkDataPacket

原需求：

> 定义 `NetworkDataPacket`：基于 MessagePack 的可序列化模型。

当前状态：**已满足**

当前已有：

```csharp
[MessagePackObject]
public class NetworkDataPacket
{
    [Key(0)] public long Timestamp { get; set; }
    [Key(1)] public int ChannelId { get; set; }
    [Key(2)] public double[] Data { get; set; } = Array.Empty<double>();
    [Key(3)] public int ActualLength { get; set; }
}
```

建议增强：

为了正式分布式链路，建议补充：

```csharp
public Guid PacketId { get; set; }
public double SampleRate { get; set; }
public string SourceNode { get; set; }
```

这样便于 ACK 对应、丢包统计和跨机定位。

## 6. 后端服务对照

### 6.1 NetworkSenderNode 作为 IPipelineNode

原需求：

> 实现 `NetworkSenderNode`：作为 `IPipelineNode` 插入流水线末端。

当前状态：**部分满足**

当前已有：

- `NetworkSenderNode : BasePipelineNode`
- `BasePipelineNode : BackgroundService, IPipelineNode`
- `Program.cs` 中注册了 `NetworkSenderNode` 为 `IPipelineNode`

当前不足：

- 当前 `sender` 模式为了演示，管线只有 `NetworkSenderNode`，不是稳定插在真实采集/处理流水线末端。
- 当前测试发送依赖手动生成 `_pendingTestPacket`，不是完全由流水线数据驱动。

建议修改：

正式模式保留两种入口：

1. 调试入口：手动生成正弦波并发送。
2. 生产入口：流水线数据进入 `NetworkSenderNode.OnProcess()`，由路由表决定是否转发。

推荐管线：

```text
AcquisitionService -> DataPushService -> NetworkSenderNode -> StorageService
```

或根据业务需要：

```text
AcquisitionService -> NetworkSenderNode
```

但需要明确配置来源，而不是写死在 `Program.cs`。

### 6.2 异步 TCP + 高性能 Buffer 队列

原需求：

> 实现异步 TCP 发送逻辑：通过高性能 Buffer 队列，将 `RefCountBuffer` 序列化并推送至远端。

当前状态：**未满足 / 部分满足**

当前已有：

- 使用 TCP 发送。
- 使用 MessagePack 序列化。
- 加了长度头解决 TCP 粘包/拆包。

当前不足：

- 当前发送是同步写入：

```csharp
stream.Write(...)
ReadFrame(...)
```

- 没有独立的发送队列。
- 没有后台发送 worker。
- 没有基于 `RefCountBuffer` 的发送生命周期管理。
- 当前发送时把 `RawDataPacket.Data` 直接放进 `NetworkDataPacket`，不是对 `RefCountBuffer` 做队列化转发。

建议修改：

1. 新增发送队列模型：

```csharp
public sealed class OutboundNetworkPacket : IDisposable
{
    public NetworkRoute Route { get; init; }
    public RefCountBuffer<RawDataPacket> Buffer { get; init; }
}
```

2. 每个远端节点维护一个 Channel 队列：

```csharp
Channel<OutboundNetworkPacket> _sendQueue;
```

3. `OnProcess()` 只做快速路由匹配和入队：

```csharp
refBuffer.Retain();
await queue.Writer.WriteAsync(packet, ct);
```

4. 后台 worker 异步发送：

```csharp
await stream.WriteAsync(...)
await stream.FlushAsync(...)
```

5. 发送完成或失败后释放引用：

```csharp
packet.Dispose();
```

6. 统计发送字节、成功包、失败包、ACK 包。

## 7. 联动需求对照

原需求：

> 当网络断开时，通过 `IMessenger` 发送“网络异常”信号，让“工程管理应用”在 UI 上显示红灯警告。

当前状态：**未满足**

当前已有：

- 使用 `IDistributedFeedbackService` 弹窗反馈网络异常。

当前不足：

- 没有使用 `IMessenger` 发送“网络异常”消息。
- `AppMessenger` 当前只是打印控制台，不是真正的消息分发。
- 工程管理应用尚未接入该消息，也没有红灯状态。

建议修改：

1. 定义网络异常消息：

```csharp
public record NetworkFaultMessage(
    string NodeName,
    string Endpoint,
    string Error,
    DateTime OccurredAt);
```

2. 在 `NetworkSenderNode` 连接失败或发送失败时调用：

```csharp
_messenger.Send(new NetworkFaultMessage(...));
```

3. 改造 `AppMessenger` 为真正可订阅的消息总线：

```csharp
void Send<T>(T message);
IDisposable Subscribe<T>(Action<T> handler);
```

4. 工程管理应用订阅 `NetworkFaultMessage`，在 UI 上显示红灯。

## 8. 工业级约束对照

### 8.1 Win7 渲染兜底

状态：**部分满足**

当前已有：

- Shell 中有 Windows 7 兼容渲染配置：

```csharp
Win32CompositionMode.RedirectionSurface
```

当前不足：

- 尚未形成测试记录，证明 UI 在 Win11 开发机测试且未使用 Win10 特有 API。

建议：

- 在测试报告中增加 Win7/Win11 兼容检查项。
- 禁止引入 WinUI、Windows 10 专属 API。

### 8.2 DI 注册一致性

状态：**未满足**

原要求：

> 各插件必须在各自的 `AddXXXModule` 扩展方法中完成所有 Service 和 IPipelineNode 的注册。

当前情况：

- 服务注册集中在 `Program.cs`。
- 没有 `AddDistributionModule`。
- 插件项目没有独立模块注册扩展。

建议：

新增：

```csharp
public static class DistributionModule
{
    public static IServiceCollection AddDistributionModule(this IServiceCollection services)
    {
        services.AddSingleton<NetworkSenderNode>();
        services.AddSingleton<NetworkReceiverNode>();
        services.AddSingleton<INetworkService>(sp => sp.GetRequiredService<NetworkSenderNode>());
        services.AddSingleton<IPipelineNode>(sp => sp.GetRequiredService<NetworkSenderNode>());
        services.AddSingleton<IPipelineNode>(sp => sp.GetRequiredService<NetworkReceiverNode>());
        return services;
    }
}
```

然后 `Program.cs` 改为：

```csharp
services.AddDistributionModule();
```

### 8.3 异常处理

状态：**部分满足**

当前已有：

- `BasePipelineNode.ExecuteAsync()` 有 `try-catch`。
- `NetworkSenderNode.SendPacket()` 内部有 `try-catch`。
- `NetworkReceiverNode.HandleClientAsync()` 内部有 `try-catch`。

当前不足：

- `RunAsSourceAsync()` 中 `TcpListener.Start()` 如果端口占用，异常会被基类捕获，但没有日志接口记录，也没有 UI 明确链路状态。
- 发送 worker 尚未拆分，后续实现异步队列后需要每个 worker 自己保护异常。

建议：

- 保留现有 `try-catch`。
- 引入 `ILogger<NetworkSenderNode>` / `ILogger<NetworkReceiverNode>`。
- Socket 异常同时写日志、更新链路状态、发送 `NetworkFaultMessage`。

### 8.4 日志埋点

状态：**未满足**

当前已有：

- `PipelineOrchestrator` 使用 `ILogger`。
- 其他大量位置使用 `Console.WriteLine` 或前端反馈服务。

当前不足：

- 网络路由更新、连接成功、发送成功、接收成功、异常等未统一调用基础设施日志接口记录 `Info` / `Warning` / `Error`。

建议：

1. 注入：

```csharp
ILogger<NetworkSenderNode>
ILogger<NetworkReceiverNode>
```

2. 关键操作记录：

```csharp
_logger.LogInformation("网络路由已更新，规则数: {Count}", routeCount);
_logger.LogInformation("已连接接收端: {Endpoint}", endpoint);
_logger.LogWarning("发送失败: {Endpoint}, {Error}", endpoint, ex.Message);
_logger.LogError(ex, "网络发送异常");
```

## 9. 推荐整改路线

### 阶段一：补齐契约和解耦

目标：让 UI 不再直接依赖 `NetworkSenderNode`。

任务：

1. 新增 `INetworkService`。
2. 扩展 `NetworkRoute`，支持节点名称和通道范围。
3. `NetworkSenderNode` 实现或委托实现 `INetworkService`。
4. `NetworkConfigViewModel` 改为依赖 `INetworkService`。

### 阶段二：实现正式映射表 UI

目标：满足“计算节点映射表”要求。

任务：

1. 将按钮式 UI 改成 `DataGrid`。
2. 支持新增/删除/编辑多条路由。
3. 支持连接/断开按钮。
4. 显示链路状态、MBps、丢包率。

### 阶段三：实现异步发送队列

目标：满足高吞吐分布式转发要求。

任务：

1. 每个目标节点建立发送队列。
2. `OnProcess()` 仅负责匹配路由和快速入队。
3. 后台 worker 负责异步 TCP 发送。
4. 对 `RefCountBuffer` 执行严格 `Retain/Dispose`。
5. 统计发送字节数、成功包、失败包、ACK 包。

### 阶段四：联动工程管理应用

目标：满足网络异常红灯告警要求。

任务：

1. 扩展 `IMessenger`，支持订阅。
2. 定义 `NetworkFaultMessage`。
3. 网络异常时发送全局消息。
4. 工程管理应用订阅消息并显示红灯。

### 阶段五：工业级验收增强

目标：满足共同规约。

任务：

1. 增加 `AddDistributionModule`。
2. 全部关键操作改用 `ILogger`。
3. 增加 Win7/Win11 兼容测试记录。
4. 增加链路断开、接收端未启动、端口占用、连续发送压力测试。

## 10. 建议最终验收清单

| 验收项 | 通过标准 |
| --- | --- |
| 计算节点映射表 | 可配置节点名、IP、端口、通道范围 |
| 路由更新 | 调用 `INetworkService.UpdateRoutingTable(List<NetworkRoute>)` |
| 手动连接/断开 | UI 有连接/断开指令，并更新链路状态 |
| 实时流量 | UI 显示每条链路 MBps |
| 丢包统计 | UI 显示发送包、ACK 包、失败包、丢包率 |
| MessagePack 模型 | `NetworkDataPacket` 可序列化并包含必要元数据 |
| Sender 管线节点 | `NetworkSenderNode` 可作为 `IPipelineNode` 插入正式管线 |
| 异步队列 | 发送不阻塞管线主处理路径 |
| 网络异常联动 | 通过 `IMessenger` 发出 `NetworkFaultMessage` |
| 红灯告警 | 工程管理应用收到消息后显示网络异常状态 |
| DI 模块化 | 通过 `AddDistributionModule` 注册 |
| 日志 | 关键操作写入 `ILogger` 的 Info/Warning/Error |
